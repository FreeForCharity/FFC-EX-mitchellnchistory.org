# Deployment Guide

This document explains how the Mitchell County Historical Society site is deployed to GitHub Pages.

## Overview

The site is a statically exported Next.js application.

- Production URL: https://mitchellnchistory.org
- Hosting: GitHub Pages
- Custom domain file: `public/CNAME`
- Custom domain value: `mitchellnchistory.org`
- Deployment workflow: `.github/workflows/deploy.yml`
- Build output: `out/`

## Architecture

The app uses static export mode in `next.config.ts`:

```ts
const nextConfig = {
  output: 'export',
  images: {
    unoptimized: true,
  },
  trailingSlash: true,
  basePath: process.env.NEXT_PUBLIC_BASE_PATH || '',
  assetPrefix: process.env.NEXT_PUBLIC_BASE_PATH || '',
}
```

Static export makes the site compatible with GitHub Pages. Images are unoptimized because the Next.js image optimizer is not available for static exports.

## Custom Domain and Base Path

The live site uses the custom domain `mitchellnchistory.org`, so production builds should use an empty base path.

The deploy workflow handles this automatically:

1. It calls the GitHub Pages API for this repository.
2. If Pages has a custom domain configured, it sets `NEXT_PUBLIC_BASE_PATH` to an empty string.
3. If Pages does not have a custom domain configured, it sets `NEXT_PUBLIC_BASE_PATH` to `/${repo_name}` for GitHub Pages subpath hosting.

This means contributors should not hard-code an old template base path such as `/FFC_Single_Page_Template`.

## Automated Deployment

Deployments run on pushes to `main` and can also be started manually from the Actions tab.

The deploy workflow performs these jobs:

1. `test`
   - checks out the repository
   - installs Node.js 20
   - runs `npm ci`
   - runs `npm run format:check`
   - runs `npm run lint`
   - runs `npm test`
   - installs Playwright Chromium
   - runs `npm run build`
   - runs `npm run test:e2e`

2. `build`
   - checks out the repository
   - installs dependencies
   - detects the correct Pages base path
   - runs `npm run build`
   - uploads `out/` as a Pages artifact

3. `deploy`
   - deploys the Pages artifact with `actions/deploy-pages`
   - runs a post-deploy smoke check against the deployed URL

4. `notify-failure`
   - opens or comments on an incident issue if production deployment fails

## Manual Local Verification

Install dependencies:

```bash
npm install
```

Run checks:

```bash
npm run format:check
npm run lint
npm test
npm run build
```

Preview the production build:

```bash
npm run preview
```

Open http://localhost:3000.

## Building for the Custom Domain

For the live custom domain, leave `NEXT_PUBLIC_BASE_PATH` unset or empty:

```bash
npm run build
```

## Testing a GitHub Pages Subpath Build

Only use a subpath when testing Pages without the custom domain:

```bash
NEXT_PUBLIC_BASE_PATH=/FFC-EX-mitchellnchistory.org npm run build
```

On Windows PowerShell:

```powershell
$env:NEXT_PUBLIC_BASE_PATH='/FFC-EX-mitchellnchistory.org'
npm run build
```

## GitHub Pages Settings

In repository settings:

- Pages source should use GitHub Actions.
- The custom domain should be `mitchellnchistory.org`.
- HTTPS should be enforced.

The checked-in `public/CNAME` file should match the active GitHub Pages custom domain.

## DNS Notes

For an apex GitHub Pages custom domain such as `mitchellnchistory.org`, configure DNS with GitHub Pages records according to GitHub's current documentation.

Typical apex records use GitHub Pages A records:

```text
185.199.108.153
185.199.109.153
185.199.110.153
185.199.111.153
```

If a `www` subdomain is used, configure it as a CNAME to the organization's GitHub Pages host.

DNS changes can take time to propagate. GitHub Pages may also need a few minutes to provision or renew HTTPS certificates.

## Troubleshooting

### Images or CSS return 404

Check whether the build used the correct `NEXT_PUBLIC_BASE_PATH`.

- Custom domain: empty base path
- Repo subpath: `/${repo_name}`

Also confirm image paths use `assetPath()` where needed.

### A route returns 404

Confirm the route is statically generated by Next.js and included in the exported `out/` directory. The app uses `trailingSlash: true`, so routes are exported as directory `index.html` files.

### Deploy workflow fails

Open the failing run in GitHub Actions and inspect the failed job:

- `test` failures are usually formatting, lint, unit test, build, or Playwright failures.
- `build` failures may include Pages API/base path or static export failures.
- `deploy` failures may include GitHub Pages artifact or environment issues.
- smoke check failures usually mean the deployed site URL, route, or asset did not respond as expected.

## Rollback

Preferred rollback is to revert the problematic commit and let the normal deploy workflow publish the previous behavior:

```bash
git revert <commit-sha>
git push origin main
```

For urgent incidents, use GitHub Actions logs and the automatically created deployment incident issue to coordinate the fix.
